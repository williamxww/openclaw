## 1. 编排体系概览

OpenClaw 的流程编排有三个层次：

```
┌─────────────────────────────────────────────────────────┐
│  Lobster（固定流程）                                      │
│  extensions/lobster/                                     │
│  YAML/JSON pipeline → 类型化 JSON envelope + 审批恢复    │
├─────────────────────────────────────────────────────────┤
│  子 Agent 系统（动态编排）                                │
│  src/agents/subagent-*.ts                               │
│  父 Agent 动态 Spawn 子 Agent，树状协作                   │
├─────────────────────────────────────────────────────────┤
│  ACP 协议（跨进程通信）                                   │
│  src/acp/                                               │
│  IDE ↔ Gateway 的标准化 Agent 通信协议                   │
└─────────────────────────────────────────────────────────┘
```

**三种 Agent 间通信工具：**

| 工具 | 用途 | 等待结果 |
|------|------|---------|
| `sessions_spawn` | 启动新子 Agent 执行任务 | 否（异步，通过 announce 通知） |
| `sessions_yield` | 结束当前 turn，等待子 Agent 完成 | 是（下一 turn 收到结果） |
| `sessions_send` | 向已有 session 发消息 | 可选（`timeoutSeconds` 控制） |

### 4.3 Lifecycle 事件监听

**文件：** `src/agents/subagent-registry.ts:957`

```typescript
function ensureListener() {
  listenerStop = onAgentEvent((evt) => {
    if (evt.stream !== "lifecycle") return;
    const entry = subagentRuns.get(evt.runId);
    if (!entry) return;

    if (phase === "start") {
      clearPendingLifecycleError(evt.runId);
      entry.startedAt = evt.data.startedAt;
    }
    if (phase === "error") {
      // 15s grace period，等待可能的重试
      schedulePendingLifecycleError({ runId, endedAt, error });
    }
    if (phase === "end") {
      if (evt.data.aborted) {
        // 15s grace period，等待可能的自动重试
        schedulePendingLifecycleTimeout({ runId, endedAt });
        return;
      }
      if (evt.data.yielded === true) {
        markSubagentRunPausedAfterYield({ entry, endedAt });
        return;
      }
      // 正常完成
      completeSubagentRun({ runId, outcome: { status: "ok" }, sendFarewell: true });
    }
  });
}
```

## 7. ACP 协议（Agent Communication Protocol）

ACP 是 IDE（Kiro/VS Code 等）与 OpenClaw Gateway 之间的标准化通信协议，基于 `@agentclientprotocol/sdk`。

### 7.1 架构

```
IDE（Kiro/VS Code）
  │  ACP 协议（ndJSON over stdio）
  ↓
ACP Server（openclaw acp）
  src/acp/server.ts
  │  WebSocket
  ↓
OpenClaw Gateway
  src/gateway/
```

ACP Server 是一个独立进程，通过 **stdio** 与 IDE 通信，通过 **WebSocket** 与 Gateway 通信。

```typescript
  // 1. 连接 Gateway WebSocket
  const gateway = new GatewayClient({ url, token, onEvent, onHelloOk, onClose });
```

```
  ↓
resolveGatewayDecisionFromPermissionOutcome() → "allow-once" | "deny"
  ↓
gateway.request("exec.approval.resolve", { id, decision })
```

- **断线重连**：Gateway 断线后，通过 `agent.wait` 轮询恢复 pending prompts

## 8. Lobster 固定流程

Lobster 是一个**类型化 JSON-first 的本地工作流运行时**，支持可恢复的审批流程。作为可选插件集成到 OpenClaw，不修改核心代码。

### 8.1 核心概念

```
Lobster Pipeline（.lobster / .yaml / .yml / .json）
  ↓ 定义固定步骤序列
Lobster Runtime（@clawdbot/lobster/core）
  ↓ 在进程内执行
LobsterEnvelope（类型化结果）
  ├─ ok=true, status="ok"           → 流程完成
  ├─ ok=true, status="needs_approval" → 需要人工审批，挂起等待
  └─ ok=false                       → 流程失败
```

### 8.2 lobster 工具

**文件：** `extensions/lobster/src/lobster-tool.ts:212`

```typescript
export function createLobsterTool(api: OpenClawPluginApi, options?: LobsterToolOptions) {
  return {
    name: "lobster",
    parameters: Type.Object({
      action: Type.Unsafe<"run" | "resume">({ type: "string", enum: ["run", "resume"] }),
      pipeline: Type.Optional(Type.String()),   // pipeline 名称或文件路径
      argsJson: Type.Optional(Type.String()),   // JSON 参数
      token: Type.Optional(Type.String()),      // 恢复 token
      approvalId: Type.Optional(Type.String()), // 审批 ID
      approve: Type.Optional(Type.Boolean()),   // 审批决策
      // ManagedFlow 参数
      flowControllerId: Type.Optional(Type.String()),
      flowGoal: Type.Optional(Type.String()),
      flowId: Type.Optional(Type.String()),
      flowExpectedRevision: Type.Optional(Type.Number()),
    }),
    execute: async (_id, params) => {  // lobster-tool.ts:243
      if (action === "run") {
        const flowParams = parseRunFlowParams(params);
        if (flowParams) {
          // ManagedFlow 模式：带状态追踪
          return resolveManagedFlowToolResult(
            await runManagedLobsterFlow({ taskFlow, runner, ... })
          );
        }
        // 直接模式：运行 pipeline
        const envelope = await runner.run({ action: "run", pipeline, argsJson, cwd, timeoutMs });
      }
      if (action === "resume") {
```

### 8.3 两种执行模式

#### 直接模式（无状态追踪）

```typescript
// Agent 调用：

// 内部流程：
runner.run({ action: "run", pipeline: "deploy-check", ... })
  → @clawdbot/lobster/core 执行 pipeline
  → 返回 LobsterEnvelope

// 若 needs_approval：

  → runner.run({ action: "resume", token, approve: true })
```

#### ManagedFlow 模式（带状态追踪）

**文件：** `extensions/lobster/src/lobster-taskflow.ts:163`

```typescript
export async function runManagedLobsterFlow(
  params: RunManagedLobsterFlowParams,
): Promise<ManagedLobsterFlowResult> {
  // 1. 创建 flow 记录（持久化到 task registry）
  const flow = params.taskFlow.createManaged({
    controllerId: params.controllerId,
    goal: params.goal,
    currentStep: params.currentStep ?? "run_lobster",
  });

  // 2. 运行 Lobster pipeline
  const envelope = await params.runner.run(params.runnerParams);

  // 3. 根据结果更新 flow 状态（lobster-taskflow.ts:129）
  const mutation = applyEnvelopeToFlow({ taskFlow, flow, envelope, waitingStep });
  // envelope.status === "needs_approval" → taskFlow.setWaiting()
  // envelope.ok === true               → taskFlow.finish()
  // 失败                               → taskFlow.fail()
}
```

**ManagedFlow 状态机：**

```
created
  ↓ runManagedLobsterFlow()
running (currentStep: "run_lobster")
  ├─ ok → finished
  ├─ needs_approval → waiting (currentStep: "await_lobster_approval")
  │     ↓ resumeManagedLobsterFlow()
  │   running (currentStep: "resume_lobster")
  │     ├─ ok → finished
  │     └─ needs_approval → waiting（再次挂起）
  └─ error → failed
```

### 8.4 Runner 实现

**文件：** `extensions/lobster/src/lobster-runner.ts`

```typescript
export function createEmbeddedLobsterRunner(options?: { loadRuntime? }): LobsterRunner {
  let runtimePromise: Promise<EmbeddedToolRuntime> | undefined;
  return {
    async run(params) {
      // 懒加载 @clawdbot/lobster/core（进程内，非子进程）
      runtimePromise ??= loadEmbeddedToolRuntimeFromPackage();
      const runtime = await runtimePromise;

      return await withTimeout(params.timeoutMs, async (signal) => {
        const ctx = createEmbeddedToolContext(params, signal);

        if (params.action === "run") {
          const filePath = await detectWorkflowFile(pipeline, cwd);
          if (filePath) {
            // 文件路径模式：直接加载 .lobster/.yaml 文件
            return runtime.runToolRequest({ filePath, args, ctx });
          }
          // pipeline 名称模式
          return runtime.runToolRequest({ pipeline, ctx });
        }

        // resume 模式
        return runtime.resumeToolRequest({ token, approvalId, approved, ctx });
      });
    },
  };
}
```

**安全约束：**
- `resolveLobsterCwd()`：cwd 必须是相对路径，不能逃出 gateway 工作目录
- `createLimitedSink(maxStdoutBytes)`：stdout/stderr 有字节上限（默认 512KB）
- `withTimeout(timeoutMs)`：带 AbortSignal 的超时控制（默认 20s）

### 4.2 TaskFlow 是什么

TaskFlow 是 openclaw 内置的**跨 session 任务跟踪单元**，定义在 `src/tasks/task-flow-registry.types.ts`。它不是消息，也不是事件队列，而是一个有状态的工作项，贯穿任务从创建到完成的完整生命周期。

#### 核心数据结构

```typescript
type TaskFlowRecord = {
  flowId: string;
  syncMode: "task_mirrored" | "managed"; // managed = 外部系统驱动；task_mirrored = 跟随子任务状态
  ownerKey: string;       // 绑定的 agent session key
  controllerId?: string;  // 创建方标识（如 "webhooks/hr-leave-events"）
  revision: number;       // 乐观锁版本号，每次状态变更递增
  status: TaskFlowStatus;
  notifyPolicy: "done_only" | "state_changes" | "silent";
  goal: string;           // 任务意图描述，agent 读取后决定如何处理
  currentStep?: string;   // 当前执行到哪一步（可选，供外部系统跟踪）
  stateJson?: JsonValue;  // 任意业务状态（如请假单详情）
  waitJson?: JsonValue;   // 等待中的上下文（如等待审批的信息）
  blockedTaskId?: string; // 阻塞本 flow 的子任务 id
  blockedSummary?: string;
  cancelRequestedAt?: number;
  createdAt: number;
  updatedAt: number;
  endedAt?: number;
};
```

#### 状态机

```
                  ┌─────────────────────────────────────────┐
                  │                                         │
         create_flow                                        │
              │                                             │
              ▼                                             │
           queued ──── agent 开始处理 ──→ running           │
                                            │               │
                              ┌─────────────┼──────────┐    │
                              ▼             ▼          ▼    │
                           waiting       blocked    succeeded / failed / cancelled / lost
                              │             │
                    resume_flow /      blockedTask
                    外部系统恢复        完成后自动
                              │         恢复
                              └────→ running
```

| 状态 | 含义 |
|------|------|
| `queued` | 已创建，等待 agent 处理 |
| `running` | agent 正在处理 |
| `waiting` | 暂停，等待外部输入（如人工审批） |
| `blocked` | 被子任务阻塞，子任务完成后自动恢复 |
| `succeeded` | 正常完成 |
| `failed` | 执行失败 |
| `cancelled` | 已取消 |
| `lost` | 超时或异常丢失 |

#### 通知策略（notifyPolicy）

控制 TaskFlow 状态变化时是否向 agent 发送通知：

| 值 | 行为 |
|----|------|
| `done_only` | 仅在终态（succeeded / failed / cancelled）时通知 |
| `state_changes` | 每次状态变更都通知 |
| `silent` | 不通知，agent 自行轮询或由子任务驱动 |

#### 两种同步模式

**`managed`（外部驱动）**：由外部系统（如 webhooks 插件）显式调用 `create_flow` / `set_waiting` / `resume_flow` / `finish_flow` / `fail_flow` 控制状态流转。适合业务系统主动推送事件的场景。

**`task_mirrored`（任务镜像）**：TaskFlow 状态自动跟随其关联子任务的状态。子任务完成 → flow 自动 succeeded；子任务失败 → flow 自动 failed。适合 agent 自主发起子任务、外部系统只需查询进度的场景。

#### 乐观锁（revision）

每次状态变更操作都需要传入 `expectedRevision`，与当前 `revision` 不匹配则返回 `revision_conflict`，防止并发写冲突：

```json
{
  "action": "resume_flow",
  "flowId": "flow-abc123",
  "expectedRevision": 2,   // 必须与当前 revision 一致
  "currentStep": "notify_employee"
}
```

#### 子任务（runTask）

TaskFlow 可以挂载子任务（`run_task` action），子任务由 agent 或 ACP runtime 执行。子任务状态汇总到 `taskSummary`，外部系统可通过 `get_task_summary` 查询整体进度：

```json
{
  "action": "run_task",
  "flowId": "flow-abc123",
  "runtime": "subagent",
  "task": "核查财务数据并生成风险报告",
  "agentId": "finance-expert",
  "notifyPolicy": "done_only"
}
```

#### 持久化与跨 session 存活

TaskFlow 持久化到 SQLite（`task-flow-registry.store.sqlite.ts`），gateway 重启后自动恢复。保留期为 7 天（终态后），过期自动清理。这是它与普通消息的本质区别：**TaskFlow 不依赖 session 存活，session 重启后 flow 依然在**。
